nvm的安装

前言

在运行一个demo项目的时候,发现项目需要nodejs v6+的运行环境,而本机的环境是v4,
但是自己的项目都是在v4的环境下开发的.不想太贸然将全局的node升级到最新,
所以就需要安装 一个nvm (nodejs version management)

安装

本安装环境都是在mac下

通过curl安装

然后在 ~/.bash_profile 中加上

export NVM_DIR=”$HOME/.nvm”
[ -s “$NVM_DIR/nvm.sh” ] && . “$NVM_DIR/nvm.sh” # This loads nvm

目的是为了在每次打开bash的时候,会在环境变量中自动添加nvm

完毕之后运行nvm 查看能看到帮助信息说明安装成功,
如果提示,

nvm:command not found

可能是缺少~/.bash_profile 文件,运行 touch ~/.bash_profile 然后重新走一遍.

如果依然不行,尝试在bash_profile文件中添加一行 source ~/.bashrc后再尝试,

安装好后,运行

查看可以安装的node版本

安装指定版本

运行

正常情况下就可以安装成功

然后使用命令切换当前使用的版本

本来一切都应该如此简单美好
但是….
偏偏遇到了错误…

遇到坑

在执行install命令的时候,发出了错误(我是先安装4.6.1)

看错误的意思是在做签名比对的时候失败了.二进制文件下载失败
然后开始下载源码,然后下载源码的时候也挂了.
期初以为是网络问题.但是切换各种网络都是一样的结果.

这里的100%的进度不是一点点加载的 ,是瞬间出来,感觉其实根本没有发生下载直接发生了错误

去看过官方的issue,确实有人有类似的问题,他们提供的解决思路

  • 如果网络设置了代理,会导致这种错误,可以尝试去掉代理
  • 虚拟机的环境下也会导致失败

然后默默的干掉了代理,信心满满,原来真的是网络问题.重新跑了一次install,
OMG 还是一样的错误…

google的许久都是没有结果.

但是在不停的尝试中发现,
不用mac自带的终端,使用iterm2 去执行install的进度条是会一点点加载的,说明确实有在下载,
然后下载到一定的时候依然发生错误

可以看出来,其实就是去下载node-v4.6.1-darwin-x64.tar.gz的包,
然后放在/Users/zhangjunbin/.nvm/.cache/bin/node-v4.6.1-darwin-x64/下
于是就有了如下的尝试

去官网下载对应版本的对应文件,放到如上所示的目录(失败的情况下该目录需要自己创建)

当完成了bin的文件,丢进去后 checksum的错误没有了.但是还是失败(需要再下一次source,/src)
再下载一次相应的文件
丢到了/Users/zhangjunbin/.nvm/.cache/src/下
再运行一次 install

就安装成功了 ^_^

你不知道的DOMS

今天了解到一个新的知识,然后他又有一个非常熟悉的名字叫做DOMS,乍一看,太熟悉不过了,Document Object Model 嘛~
但是今天的主题不是技术

总描述不出的那个词

健身多年,对健身的知识的了解也是零零散散,懂一些,但是都形不成系统的理论,更有些东西明明知道肯定会有一个专业的学术名词去定义,
但是自己无法准确的描述出来,好多次尝试去百度,但是得到的结果都不是想要的.直到上个礼拜某一天,在一片文章里面看到一个名词

DOMS(delayed onset muscle soreness),延迟性肌肉疼痛,

瞬间我就意识到,这个就是困扰我好久但是就是将不出来的那个”学术名词”.经过一番百度谷歌,对此物做一记录.

延迟性肌肉疼痛

延迟性肌肉疼痛,也称为肌发热,是在非经常性运动或者剧烈运动之后的几小时到几天内肌肉感到酸痛或疼痛,
这种酸痛运动后在24小时到72小时会达到高峰

特征

在发生酸痛的肌肉会使肌肉迟钝,伴随着压痛,但是疼痛只有在肌肉发生拉伸或者收缩,静止的肌肉不会感到疼痛,
在百科上,这称作”肌肉机械痛觉过敏”.

原因

产生这种酸痛的最主要原因是因为肌肉做离心(延展)运动而产生的

离心运动:(Eccentric Muscle Action)肌肉收缩力量小于外在负荷的阻抗力量,肌肉被拉长,此时肌肉产生的力量小于外在负荷伸展肌腱的力量.举重训练时动作向下阶段为离心运动,在标准运动中,肌肉离心运动是为了抵抗负荷向下的重力加速度.所有希望产生离心运动时,必须稳定的阻抗向下负荷而非速度放下.

而同心(缩短)运动基本不会产生酸痛.

同心运动:(Concentric Muscle Action)肌肉收缩力量大于外在负荷的阻抗力量时时,肌肉缩短,肌肉产生的力量大于外在负荷伸展肌腱的力量,游泳,单车等肌肉都是做向心运动

等距(静态)运动产生的酸痛较少,

等距运动:(Isometric Muscle Action)肌肉收缩力量等于外在负荷的阻抗力量,肌肉长度维持不变,此时肌肉产生的力量等于外在阻抗伸展肌腱的量.仰卧起坐,上身卷举45°然后固定不动时,肌肉等长收缩相对的,上半身向上卷起时,为向心运动;上半身向下放平时,则为离心运动.

在之前的知识体系中,我总以为引起酸痛是因为肌肉中乳酸的堆积.但是事实上血液中的乳酸并未增加
(长久不动或者超负荷的运动会使得肌肉的血液中堆积大量的乳酸,也会产生酸痛的感觉)

治疗与预防

对于疼痛的治疗,大家的直觉都是休息,
而延迟性肌肉酸痛的治疗却是反直觉的,
即增加血液流向肌肉,继续锻炼,使用更轻的重量,多组数,低强度的拉伸.对缓解疼痛都有帮助,也有助于提高肌肉的痛阈和耐痛.
在实际的健身中也会有明显的感觉.如果越是怕痛越不敢动,往往只能等待痛疼消失.
而尝试的去做低负荷的运动,按摩,或者热水浴,都会有助于缓解疼痛

目前并没有非常有效的预防方法,但是实践证明.
适度的拉伸(过度拉伸也会导致疼痛),有效的热身,和有计划的健身,都能够避免或者减少DOMS的发生

MarkDown 使用指南

概述

Markdown 语法的目标是：成为一种适用于网络的书写语言。
Markdown 不是想要取代 HTML，甚至也没有要和它相近,它的语法种类很少,只对应 HTML 标记的一小部分
Markdown 的理念是,能让文档更容易读写与修改.不在 Markdown 涵盖范围之内的标签，都可以直接在文档里面用 HTML 撰写。不需要额外标注这是 HTML 或是 Markdown；只要直接加标签就可以了.

由于Markdown最终会被转成html标签.所以在文本需要描述一个html标签,
遇到<是需要书写为&lt;
而 & 需要书写为 &amp;

  • *,-,+ 3个符号效果都一样,被称为Markdown符号
  • 空白行表示另起一个段落
  • `表示行内代码,tab 表示 代码段

标题设置

一级标题,类似于<h1>,在文字下方使用===标注 (任意数量的=)
二级标题,类似于<h2>,在文字下方使用---标注
示例:

一级标题
===
二级标题
---

一级标题

二级标题

或者:

文字

  • 斜体: 将需要设置为斜体的文字两端用1个 * 或者 - 包裹起来,就像这样子写:*斜体* 斜体
  • 粗体: 将需要设置为粗体的文字两端用2个 * 或者 - 包裹起来,就像这样子写:**斜体** 斜体

引用

在段落前使用 > 表示该段文字是一段引用,就像:

有序与无序列表看起来一样,其实无序列表被解析为ul,而有序列表被解析为ol

Git 常用命令

git 命令行操作
git add -i(interactive) 交互式暂存

git remote add [库别名] [库地址]
git remote rm [库别名]
git remote -v 查看
git push [库别名] [分支]
git push [库别名] [分支] –tags
git status
git cache -rm

git pull ==>
git fetch + git merge
如果需要以rebase的方式,需要使用
git pull –rebase

#分支管理
git branch 查看所有本地分支
git branch -a 查看所有分支
git branch -r 查看远程分支
git branch dev 创建分支
git checkout -b develop master 创建+切换分支
git checkout master 切换到Master分支
git checkout -filename.txt 丢弃工作区修改( - 很重要)
git banch -d bug 删除分支
git merge –no-ff develop 将develop分支合并到当前分支git

#工作区
git stash 挂起工作区.
git stash list 查看挂起的工作区
git stash pop 回复工作区内容并删除记录

#版本回退
git reset –hard HEAD^ 上一个版本
git reset –hard HEAD^^ 上两个版本
git reset –hard HEAD~100 上100个版本
git reset –hard [版本号] 指定版本号
reset之后 在指定版本后的commit会无法再log中显示,但是依然可以追踪
推荐使用revert 回退到指定版本的文件状态,可能会遇到冲突,完成冲突之后产生一个新的提交
git revert [commit*]
在发生冲突之后记得使用 –continue 继续或者使用 –quit/abort 取消回退
!!!
revert 是回退到这个版本提交之前的内容
reset 是回退到这个版本提交之后的样子

#bug修复的流程
git checkout -b [分支名] //在需要开分支的分支上
…修改,提交git
切换回主分支
git merge –no-ff [分支名]
必须带上–no-ff 为了版本演进清晰

#分支冲突解决流程

#git rebase
git checkout [分支名(在开发的,最新的)]
git rebase [分支名]

git rebase –continue 发生冲突之后,解决完继续
git rebase –abort 停止rebase

#git tag

git tag -a [tag 名称] [commit Hash] 新建tag并且需要备注
如果加上 -m ‘xxxx’ 可以直接完成备注
git tag -d [tag 名称] 删除tag 删除本地的

git push –tags 推送tags 到远端
git push origin –delete tag [tag 名称] 删除远端tag

#查看已经commit未push的commit
git cherry -v
git log [分支名] ^origin/[分支名]
